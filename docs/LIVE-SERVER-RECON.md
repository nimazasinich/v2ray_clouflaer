## 9. Why this can't be fixed by the agent from outside

There are **two simultaneous architectural conflicts** on the box, and
either fix carries operational risk that requires operator approval:

### Conflict A — port 443 ownership

xray Reality wants `:443` (it is *the* canonical port for Reality
camouflage). nginx's `cdn` site also declares `:443 ssl`. nginx loses
the bind, so its TLS termination for `cdn.*` is dead. Every CF-fronted
HTTPS request to `cdn.*:443` ends up at xray, which speaks Reality
(client expects normal TLS), so CF gets a TLS error → 525.

**Possible resolutions** (all need operator):

- Move nginx `cdn` site to a different CF-HTTPS port (e.g. `:2086`,
  `:2096`), update Cloudflare DNS or page rules accordingly, but
  `:2095/2096` are also xray-bound. There's no free CF-HTTPS port
  without giving up a Reality inbound.
- Front xray Reality with HAProxy/SNI-based routing (xray's `fallback`
  feature), so non-Reality TLS handshakes fall back to nginx. This is
  invasive and changes the auth path for 15 active users.
- Delete the nginx `cdn` site entirely, and instead have xray inbounds
  on :2083/2052/2082/2053 do their own TLS termination (add `tls`
  security with the LE cert in xray streamSettings).

### Conflict B — plaintext xray inbounds on CF-HTTPS ports

The xray inbounds on `:2083` and `:2053` (CF-HTTPS group) are configured
with `security: none`, meaning they speak plaintext WS/xhttp. Cloudflare
on its `:2083/:2053` ports always presents itself to the origin as TLS
(`Full` SSL mode). Mismatch → every request returns 525.

**Possible resolutions** (all need operator):

- Switch those xray inbounds from CF-HTTPS group ports (`:2053/:2083`)
  to CF-HTTP group ports (`:2052/:2082/:2086/:2095`). Cloudflare on
  HTTP ports speaks plain HTTP/WS to the origin; this matches.
- Or: add TLS security to those xray inbounds, pointing at the LE cert.
  Origin now speaks HTTPS, CF is happy. But xray loses some WS-perf.
