## Changes applied

### B2 — xray CDN inbound port migration

`/usr/local/etc/xray/config.json` was patched (atomically, with backup) to
move two plaintext CDN inbounds off the CF-HTTPS-group ports (where they
caused HTTP 525 due to TLS-vs-plaintext mismatch with Cloudflare in `Full`
SSL mode) to CF-HTTP-group ports (where Cloudflare proxies plain
HTTP/WebSocket to the origin):

| Tag | Old port (CF-HTTPS) | New port (CF-HTTP) |
|---|---|---|
| `cdn-vless-ws-2083` → `cdn-vless-ws-2086`     | 2083 | **2086** |
| `cdn-vless-xhttp-2053` → `cdn-vless-xhttp-8880` | 2053 | **8880** |

Validated with `xray run -test`, restarted via `systemctl restart xray`,
confirmed active.

Inbounds `cdn-trojan-ws-2052` (port 2052) and `cdn-vmess-ws-2082` (port
2082) were already on CF-HTTP-group ports and required no change.

### A1 — strip dead `:443 ssl` from nginx cdn site

`/etc/nginx/sites-available/cdn` previously declared `listen 443 ssl http2`
with cert paths `/root/cert.crt` (a self-signed cert with no SAN). nginx
could **never bind** :443 because xray Reality already owns it; the bind
attempts produced repeated `[emerg] bind() to 0.0.0.0:443 failed (98:
Address already in use)` log spam.

Replaced the file with a minimal `:80`-only site that returns a static
camouflage page + `/health` endpoint. nginx no longer fights xray for
:443.

### C — clean up apex `xray-ws` site

`/etc/nginx/sites-available/xray-ws` previously did
`proxy_pass http://127.0.0.1:8880` for the apex domain, but nothing
listened on `:8880`. After B2, port 8880 is xray xhttp — but the apex
site is for `dreammaker-groupsoft.ir`, not `cdn.*`. So instead of
re-routing to xray xhttp (wrong hostname), the site was simplified to a
camouflage `:80` page with `/health` only. CDN traffic intentionally goes
to `cdn.dreammaker-groupsoft.ir` on the dedicated xray ports.
